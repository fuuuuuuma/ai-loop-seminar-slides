ループ設計の4要素

要素① 目標を与える — /loop と /goal

  • /loop = 同じタスクを回し続けさせる指示
  • /goal = ゴールを設定し、達成まで自律実行させる指示
  • 共通点: 「1回やって終わり」ではなく「達成するまで」を明示する
「やって」ではなく「できるまでやって」が合言葉

① ゴールは定量で。優先順位もセットで

弱いゴール

「字幕をいい感じに整えて」

「いい感じ」をAIが判定できない
→ ループが終われない/暴走する

強いゴール

「1行25字以内・誤字ゼロ・全行レビュー合格まで。品質＞速度で」

数値+合格条件+優先順位
→ AIが自分で終了判定できる

① ゴール設定テンプレート

◆ 目標: ［定量的な完了条件］になるまで続ける

◆ 優先順位: ［品質 / 速度 / コスト］の順で重視

◆ 合格判定: ［誰が・何で検証するか］

◆ 報告: 達成したら［形式］で報告。途中で詰まったら質問

※特典の「/loop用プロンプト集」に記入済みサンプルを同梱します

② CLAUDE.md の構造例

# ルール（トリガー → アクション）

- 結果を伝えるとき → 結果を先に。説明は聞かれたら

- 取り消せない操作の前 → 必ず確認を取る

- 失敗パターンが出たとき → memory/*.md に追記する

# 禁止

- 認証情報のコミット / 勝手な外部公開

ポイント: 自己紹介ではなく「条件 → 行動」の契約書として書く

要素③ 検証 — リレーとループは別物

③ 合格条件を明文化する

例:「3体のレビューAI全員が合格を出すまで完成と認めない」

  • 検証役は生成役と別のエージェントに分ける（自己採点は甘くなる）
  • 観点を分ける: 正確性 / デザイン / 誤字 など1体1観点
  • 「全員合格」が出た成果物だけが人間に届く

③ Auto許可 — 人の承認で止めない

ループの最大の敵は、「許可待ち」での停止です。
夜中の2時にAIが「続けていいですか？」と聞いてきても、誰も答えられません。

  • 安全な操作は事前に包括許可しておく
  • 危険な操作だけ人間の確認に残す（次のスライド）
  • = 安全なガードレールの中で速度を最大化

③ ガードレールの引き方

AIに任せて良い（Auto許可）
  • ファイルの読み取り・検索
  • 作業フォルダ内の生成・編集
  • テスト・検証の実行
  • 下書き・レポートの作成
人間に残す（要確認）
  • 削除・上書き・force push
  • 外部への送信・公開・投稿
  • 課金が発生する操作
  • 認証情報に触れる操作

要素④ 継続条件 — 回り続ける環境を作る

どんなに賢いAI社員も、
PCがスリープしたら止まります

ループ設計の最後のピースは、モデルでもプロンプトでもなく
「物理的に回り続ける環境」です。

④ 環境の装備リスト

装備役割
デスクトップアプリ版長時間セッションの安定運用・バックグラウンド実行
Amphetamine 等Macのスリープ抑止（ループの突然死を防ぐ）
Computer Use画面操作・実機検証をAI自身に行わせる
通知（Slack / Push）完了・異常だけ人間に知らせる

④ 4つの運用パラメータ

間隔

何分おきに回すか。短すぎは無駄、長すぎは機会損失

頻度・回数

1日何回まで・何回失敗したら停止するか

リソース

PC負荷・並列数の上限。本業の作業を妨げない設計

コスト

トークン予算の上限を必ず設定。青天井にしない

実例① AI × 動画編集

無音カット → テロップ → 字幕SRT を
コマンド1発のループに

  • 素材を渡すと、編集ソフト用のカット済みデータと字幕が出てくる
  • 人間がやるのは最終チェックと微調整だけ

① Before: すべて手作業だった頃

  • 無音部分を探して1カットずつ削除
  • 字幕は聞き起こして1行ずつ手入力
  • 1本あたり数時間が当たり前

「編集が追いつかないから投稿本数を増やせない」状態でした。

① ループ設計（4要素に分解）

要素設計内容
① 目標全字幕が「1行の文字数制限内・意味の切れ目で改行」になるまで
② エージェント文字起こし役 / 改行判定役 / 組み立て役 の3分業
③ 検証レビューAIが全行チェック → 不合格行だけ差し戻し
④ 継続条件長尺動画はタイムアウト延長・並列化で完走させる

① After

編集作業の大部分が「待ち時間」に変わった

  • カット編集: 数時間 → コマンド実行後は放置
  • 字幕: 聞き起こし不要。読みやすい改行まで自動
  • 品質が「人の集中力」に左右されなくなった

① 成功のポイント

肝は生成ではなく「検証の設計」

カットしすぎ・取りこぼしを別のAIがレビューする工程を入れた瞬間、
人間の最終チェックがほぼ素通りになりました。

実例② YouTube運用

企画 → 資料 → 投稿文 → 分析 を
運用ループとして設計

  • リサーチ資料・概要欄・X告知文・note記事までAI組織が生成
  • 数値分析も定点観測のループに組み込み

② Before

  • 1チャンネル分の企画・台本・告知だけで手一杯
  • 分析は「数字を眺めるだけ」で次に活きない
  • 横展開（X・note・ショート）は構想のまま放置

② ループ設計（4要素に分解）

要素設計内容
① 目標動画1本から「資料・概要欄・X投稿・note記事」一式を毎回そろえる
② エージェントリサーチ役 / 記事執筆役 / 投稿文役 / 分析役を媒体別に分業
③ 検証ファクトチェック役が公開前に全文を検査（誇張・固有名詞）
④ 継続条件定点観測は定期実行。投稿だけは人間に残す

② After

YouTube 3チャンネルの同時運用が
1人で現実的に回るように

  • 1本の動画から各媒体への展開がほぼ自動で出揃う
  • 分析が「観察」から「次の一手の提案」に変わった

② 成功のポイント

「投稿だけは手動」というガードレール

全部任せるのが正解ではありません。
アカウントBANや誤投稿のリスクが残る工程は、あえて人間に残すのが設計です。

実例③ スライド・営業資料・LP作成
